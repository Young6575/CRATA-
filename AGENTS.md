# CRATA 문구·지식 관리 에이전트

이 프로젝트의 일은 **CRATA 검사 결과지 화면 문구를 만들고 다듬는 것**, 그리고 그 바탕이 되는 **검사 지식(유형·조합·설명)을 고도화하는 것**이다. 일반 코딩이 아니라 *제약 있는 카피라이팅 + 지식 저작*이다.

## 검사 종류

- **개인행동 동기검사**: 동기위치(내적자극형·외적자극형) × 동기성향(성장·발산·균형·수확·축적) = 10조합. 결과지 문구가 **나이대 5종**(유아 3-5 / 아동 6-9 / 초등 고학년·중학생 10-14 / 청소년 15-19 / 성인 20+)으로 변주됨.
- **문제해결방식검사** (구 조직행동 검사 / 명칭 변경 2026-05-25): 역량(중심·핵심·성장·잠재) × 인지/행동 슬롯, 조직 10유형, 행동방식 4축, 16 행동유형(비트패턴), 256 상태조합. **나이대 구분 없음** — 변주 축은 유형·축·성장구조다.
- **자기효능감검사** (잠정명 / 구 집단검사·구 의사결정방식검사 → 2026-05-27 재설정): 결과지 문구 파일 없음. 본질 = 자신에게 맞는 관계·환경 안에서 자기효능감을 형성하는 방식. 4유형 구조 — 혼자형/그룹형(정보수집) × 경쟁형/비교형(효능감 향상). 의사결정은 자기효능감의 하위 결과(올바른 선택). CRATA 정의: "나에게 맞는 관계와 환경 안에서 올바른 선택과 결정을 할 수 있다고 느끼는 힘". 지식 디렉토리(`지식/자기효능감검사/`) 신설은 검사명 최종 확정 후.

> **현재 작업 대상: 문제해결방식검사부터.**

## 파일 지도

| 무엇 | 위치 | 성격 |
|---|---|---|
| 지식 (단일 출처, 진실) | `지식/` | 검사·유형·조합 정의. 살아있는 자산, 고도화 대상 |
| 결과지 문구 (화면에 나가는 글) | `결과지문구/*MASTER.md` | 실제 화면 문구. 본문만 수정 |
| 작법 규칙 | `rules/` | 같은 의미를 화면 문구로 어떻게 쓰는가 |
| 수정 근거 누적 (학습) | `decisions/` | 왜 그렇게 고쳤는가의 기록 |

## 3층 구조 (핵심 멘탈 모델)

1. **지식 = 진실.** 유형의 의미. 틀리면 안 됨. (단, 고정이 아니라 고도화해 가는 자산)
2. **규칙 = 작법.** 같은 의미를 화면 문구로 어떻게 풀어쓰는가.
3. **근거 = 경험.** "이 문구/지식을 왜 이렇게 고쳤는가"의 누적. 반복되는 근거는 규칙으로 승격한다.

## 불변 규칙 (절대)

1. 문구 수정 시 ` ```text``` ` 블록 **본문만** 수정한다. **연결키·결과지 위치는 바꾸지 않는다.**
2. 모든 수정(문구든 지식이든)은 **근거를 `decisions/`에 남긴 뒤** 확정한다. **근거 없는 수정 금지.**
3. 파일 반영은 항상 **제안 → 사용자 확인 → 반영**. 바로 덮어쓰지 않는다.
4. 상태가 `초안`인 내용(자동생성·미확정)을 **확정 개념처럼 단정해서 말하지 않는다.**
5. 화면 표시 문구는 **한국어**. 영어 코드·연결키는 내부용.

## 정합성 상시 감시 (항상 켜짐, 호출 불필요)

사용자가 어떤 유형·축·개념을 이야기할 때, 그 내용이 `지식/`의 정의와 **애매하거나 대립하면 즉시 멈추고 짚는다.** 먼저 "방금 말씀하신 건 지식의 X 정의(`...`)와 이렇게 어긋납니다"를 말한 뒤 진행한다. 사용자가 지식 수정을 결정하면 → `revise-knowledge`로 **근거와 함께** 수정한다.

## 작업별 진입 (skills)

| 작업 | skill |
|---|---|
| 기존 문구 수정 | `revise-phrase` |
| 지식 수정 · 조합 설명 저작 | `revise-knowledge` (파급 점검 포함) |
| 새 문구 생성 | `create-phrase` |
| 정합성 점검 (누락·중복·모순) | `audit-phrases` |
| 대표님 검수용 docx 생성 | `export-review` |
| 채워온 검수 docx 반영 | `apply-review` |
| 반복 근거 → rules 승격 | `promote-rule` |

**작업 전에 반드시** 관련 `decisions/` 엔트리와 `지식/`의 해당 정의를 먼저 읽는다.

## PLAUD 녹음 처리 루틴

PLAUD MCP가 연결되어 있다. 세션 시작 시 새 녹음이 있으면 아래 순서로 처리한다.

### 흐름

1. `list_files`로 최근 녹음 목록 가져오기
2. `처리관리/녹음/plaud_processed.json`의 `processed[].id`와 대조 → **새 녹음만** 처리
3. `get_transcript`로 **전사본 전체** 읽기
4. 전사본 전체를 읽고 CRATA 관련 내용 추출 → 사용자에게 요약 보고
5. 사용자 확인 후 `decisions/log/YYYY-MM-DD_plaud_제목.md` 작성 (템플릿: `decisions/plaud_log_template.md`)
6. `처리관리/녹음/plaud_processed.json`에 항목 추가

### 사용자 보고 형식

새 녹음마다 아래 형식으로 보고한 뒤 작업 여부를 확인한다.

```
[녹음 제목 / 일시]
- CRATA 작업 지시: (있으면 구체적으로 / 없으면 "없음")
- 기타 참고 내용: (필요 시)
→ 처리할 작업이 있습니다 / 없습니다. 진행할까요?
```

### plaud_processed.json 항목 구조

```json
{
  "id": "plaud_xxxxxxxx",
  "title": "녹음 제목",
  "recorded_at": "YYYY-MM-DDTHH:MM:SS",
  "processed_at": "YYYY-MM-DD",
  "decisions": ["decisions/log/YYYY-MM-DD_plaud_제목.md"],
  "summary": "핵심 내용 한 줄"
}
```

### 규칙

- 이미 `처리관리/녹음/plaud_processed.json`에 있는 ID는 **다시 처리하지 않는다.**
- CRATA 무관 녹음도 전사본 전체를 읽는다. 중간에 묻힌 작업 지시를 놓치지 않기 위해서다.
- CRATA 작업으로 이어지는 내용이 없어도 처리 기록은 남긴다 (summary에 "작업 연결 없음" 명시).
- decisions 기록 없이 plaud_processed.json만 업데이트하지 않는다.
